# 修复清单

## 文档目的

本清单用于整理当前 `E:\复利选股` 项目的关键偏差和修复优先级，帮助后续按最小成本修正 MVP，使其尽快回到可交付状态。

适用范围：

- `backend/`
- `frontend/`

---

## 修复优先级说明

- `P1`：必须优先修复，影响结果正确性或基础可用性
- `P2`：应尽快修复，影响用户理解、交互质量或与规格一致性
- `P3`：建议修复，影响可维护性或产品可信度

---

## P1

### 1. 复利计算口径与规格不一致

问题说明：

- 当前后端实现采用的是“先加本月定投，再计算本月收益”
- 这等价于“月初定投”
- 但当前 MVP 规格中明确写的是“默认每月末定投”

影响：

- 最终资产会被系统性高估
- 总收益、收益倍数、方案横向对比结论都会受影响
- 前后文档和实际结果不一致

问题位置：

- `backend/src/main/java/com/example/compound/service/CompoundServiceImpl.java`

关键代码点：

- `currentAmount = (currentAmount + input.getMonthlyContribution()) * (1 + monthlyNetRate);`

修复建议：

- 将计算顺序改为：
  1. 先让当前资产按月收益增长
  2. 再在月末加入定投金额

建议目标逻辑：

```java
currentAmount = currentAmount * (1 + monthlyNetRate);
currentAmount = currentAmount + input.getMonthlyContribution();
```

修复后需要同步确认：

- 年度曲线数据是否仍然按年末节点输出
- `totalInvested`、`finalAmount`、`totalProfit` 是否与页面展示一致

---

### 2. 中文文案出现乱码，直接影响页面和接口可用性

问题说明：

- 当前前后端多处中文文本已出现乱码
- 包括：
  - 默认方案名
  - 表单标签
  - 按钮文案
  - 风险提示
  - 后端参数校验报错
  - 接口异常信息

影响：

- 用户无法正常理解页面
- 错误提示不可用
- 后续继续改动时容易造成更多编码污染

问题位置：

- `backend/src/main/java/com/example/compound/dto/ScenarioInput.java`
- `backend/src/main/java/com/example/compound/exception/GlobalExceptionHandler.java`
- `backend/pom.xml`
- `frontend/src/pages/ComparePage.jsx`
- `frontend/src/components/ScenarioForm.jsx`
- `frontend/src/components/ResultCard.jsx`
- `frontend/src/components/ComparisonChart.jsx`
- `frontend/src/services/api.js`

修复建议：

- 统一所有源码文件保存为 `UTF-8`
- 恢复所有中文文案为正常文本
- 检查 IDE、终端、Git 提交编码设置，避免重复污染

建议检查项：

- Java 源码编码
- JSX 文件编码
- `pom.xml` 中项目描述文本
- 终端输出字符集

验收标准：

- 页面中文全部正常显示
- 接口报错中文可读
- 默认方案名称正常

## P2

### 3. 前端结果卡片的“排名”与真实优劣无关

问题说明：

- 当前卡片 `rank` 直接使用返回数组下标
- 后端只返回 `bestScenarioName`，没有按结果排序
- 因此前端显示的“1、2、3”只是输入顺序，不是真实排名

影响：

- 用户容易误以为列表已经按收益表现排序
- “最优方案”徽标与 rank 数字可能冲突

问题位置：

- `frontend/src/pages/ComparePage.jsx`
- `frontend/src/components/ResultCard.jsx`

修复建议：

二选一即可：

1. 最简单方案  
   去掉 `rank` 展示，只保留“最优方案”高亮

2. 更完整方案  
   前端按 `realFinalAmount` 或 `finalAmount` 排序后再展示

MVP 建议：

- 优先采用方案 1，减少误导

---

### 4. 前端没有正确展示后端校验错误详情

问题说明：

- 后端校验失败时返回的是 `messages` 数组
- 前端只尝试读取 `message`

影响：

- 用户只能看到笼统错误提示
- 无法快速知道是哪个字段错了

问题位置：

- `backend/src/main/java/com/example/compound/exception/GlobalExceptionHandler.java`
- `frontend/src/services/api.js`

修复建议：

- 前端解析后端返回 JSON
- 若存在 `messages` 数组，优先拼接或展示第一条错误

建议实现：

```js
if (Array.isArray(err.messages) && err.messages.length > 0) {
  msg = err.messages[0];
}
```

可选增强：

- 多条错误按换行展示

---

### 5. 技术基线与已定方案不一致

问题说明：

- 当前后端 `pom.xml` 中 Java 版本为 `17`
- 但当前产品方案已经明确收敛到 `Java 21`

影响：

- 与既定技术方案不一致
- 后续如果按 Java 21 环境部署或扩展，会增加不必要差异

问题位置：

- `backend/pom.xml`

修复建议：

- 将 `java.version` 从 `17` 改为 `21`
- 同时确认本地 JDK 环境与 Maven 编译环境一致

## P3

### 6. 结论文案有部分是固定输出，不完全基于当前输入结果

问题说明：

- 当前 insight 中至少有部分结论是固定追加
- 没有完全根据用户实际输入进行推导

影响：

- 用户可能误以为这些结论是针对本次结果生成的
- 产品可信度会下降

问题位置：

- `backend/src/main/java/com/example/compound/service/CompoundServiceImpl.java`

修复建议：

- 保留 1 到 2 条通用教育文案可以接受
- 但至少第一条结论应严格基于当前方案间差异计算得出
- 对“费用率影响”类结论，最好判断输入方案确实存在明显费率差异后再展示

---

## 建议修复顺序

建议严格按以下顺序处理：

1. 修复所有编码乱码问题
2. 修正复利计算顺序，统一为“月末定投”
3. 修复前端错误提示解析
4. 去掉或修正错误的 rank 展示
5. 将 Java 版本调整为 21
6. 优化 insight 生成逻辑

---

## 最小验收标准

修完后，至少应满足：

- 页面中文全部正常
- 默认示例方案正常显示
- 表单报错可读
- compare 接口返回值与规格一致
- 复利计算口径与文档一致
- 页面不出现误导性的排序展示
